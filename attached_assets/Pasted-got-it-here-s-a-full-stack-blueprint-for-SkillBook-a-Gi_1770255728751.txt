# SkillBook Blueprint (GitHub for Agent Skills)

## 0) Core idea

SkillBook is a source-control + registry for “agent skills”:

* **Repository model:** Each skill lives in a repo (public/private)
* **Canonical interface:** `skill.md` is the entry contract (metadata + spec + requirements + usage + tests)
* **Validation & security pipeline:** every commit runs:

  * schema checks (does it satisfy OpenClaw expectations)
  * static checks (dangerous operations)
  * “dry-run” simulation (limited sandbox)
  * quality checks (doc completeness, determinism, tool definitions)
* **Distribution:** OpenClaw can install skills by reference:

  * `skillbook install owner/skill@v1.2.0`
  * or by commit hash

# 2) Skill format & contract

## 2.1 Repository layout (recommended)

```
/skill.md                # canonical contract (required)
/README.md               # marketing / overview (optional)
/src/                    # implementation code (optional)
/tests/                  # tests & fixtures (optional)
/schemas/                # optional json schema, tool definitions
/skill.lock              # generated dependency pin (optional)
/examples/               # usage examples, demos
/LICENSE
```

## 2.2 `skill.md` standard sections (enforced)

At minimum:

* **Metadata**

  * name, description, author, license
  * versioning strategy
  * supported platforms (win/mac/linux)
  * required runtime (node/python), min versions
  * dependencies (npm/pip/etc)
* **Tools/Functions exposed**

  * function signatures
  * required permissions
* **Setup**

  * install steps
  * environment variables / secrets required
* **Usage**

  * example prompts / commands
  * expected output shapes
* **Safety**

  * what it can access (filesystem/network)
  * rate limits / guardrails
* **Tests**

  * how to run
  * expected test fixtures
* **Changelog**

  * summary for release notes (optional but recommended)

## 2.3 Machine-readable metadata block (front matter)

Require YAML front matter in `skill.md`:

```yaml
---
skill_id: "com.skillbook.owner.skillname"
name: "Vultr Provisioner"
version: "1.2.0"
license: "MIT"
runtime:
  language: "node"
  min_version: "20"
platforms: ["linux", "windows"]
permissions:
  network: true
  filesystem: "limited"
secrets:
  - name: "VULTR_API_KEY"
capabilities:
  - "provision_vps"
  - "tag_resources"
entrypoints:
  - fn: "create_instance"
    input_schema: "./schemas/create_instance.json"
    output_schema: "./schemas/create_instance.result.json"
---
```

**Why:** the UI and validator can parse this without LLM guessing.

# 3) Validation pipeline (“OpenClaw compliance”)

## 3.1 Validation stages

### Stage A — Schema & structure

* `skill.md` exists
* YAML front matter parses
* required fields present
* versions are valid SemVer
* referenced schema files exist

### Stage B — Static analysis & safety lint

* detect forbidden patterns in code:

  * credential exfiltration primitives
  * suspicious networking to unknown endpoints
  * self-modifying behaviors outside allowed boundary
* enforce safe defaults:

  * no wildcard file deletion
  * no “exec arbitrary command” unless explicitly declared and gated
* require “permissions declaration” match what code actually does (best-effort)

### Stage C — Dependency checks

* lock dependencies, prevent typosquatting packages
* vulnerability scan (SCA)
* license compliance (MIT/Apache etc)

### Stage D — Sandbox dry-run tests

* run unit tests in isolated container/VM
* if it’s an automation skill:

  * run with fake credentials
  * intercept outbound calls using a mock proxy
* validate outputs match declared schemas

### Stage E — Review gate (optional)

* “Verified” badge requires:

  * passing pipeline
  * maintainer identity verification
  * manual review of risk surface

## 3.2 Result artifacts

Each commit gets:

* pass/fail per stage
* logs
* SBOM (software bill of materials)
* checksums
* “risk score”

# 4) System architecture

## 4.1 High-level components

1. **Web App (frontend)**

   * repo browsing, editor, PR review, pipeline status
2. **API Gateway (backend)**

   * auth, repo service, search service, PR service
3. **Git Service (core)**

   * either:

     * embed real git hosting (Gitea-like)
     * or implement “git-like” objects internally
4. **Validation Runner**

   * queue worker that runs checks in sandbox
5. **Artifact Store**

   * store tarballs, SBOMs, logs, signatures
6. **Search Index**

   * full text + metadata filters
7. **CLI**

   * git-style UX + install/publish tooling
8. **Registry**

   * versioned “skill packages” for install

## 4.2 “Use Git” vs “Rebuild Git” decision

### Option 1 (recommended): Use Git underneath

* Host repos using **Git server** (e.g., Gitea-compatible)
* SkillBook layers validation + registry + UI on top
* Pros: instant forks/branches/commits compatibility, devs already know it
* CLI can use real `git push` + SkillBook CLI for publish/install

### Option 2: Custom VCS

* Store “commit objects” (like git) in DB/blob store
* Pros: custom permissions/policy enforcement
* Cons: you’ll reinvent GitHub

**Blueprint will assume Option 1**.

# 9) Skill publishing flow (exact)

### A) Developer side

1. `skillbook init`
2. writes skeleton `skill.md` + schemas
3. dev codes `/src`, `/tests`
4. `git commit -m "feat: add create_instance"`
5. `git push`

### B) Server side (webhook)

6. Git webhook triggers `POST /webhooks/git/push`
7. API records commit
8. Validation job enqueued

### C) Runner

9. Runner checks out repo at commit hash
10. Runs validation stages A–D
11. Stores logs/SBOM/artifacts
12. Writes validation status

### D) Publish

13. Dev runs `skillbook publish --tag v1.2.0`
14. Server checks:

    * tag points to commit that passed validation
    * version in skill.md matches tag
15. Server bundles skill into tarball, signs & stores
16. Registry entry is created
17. skill is now installable by version

# 11) Security model (non-negotiable)

Skills are code. Treat as untrusted.

## Key protections

* Sandbox runner with:

  * no secrets by default
  * blocked network egress unless declared and allowed
  * limited filesystem
  * time/memory limits
* “Secrets” are never exposed to untrusted PR builds
* All release artifacts are checksummed
* Optional signing:

  * maintainers sign releases
  * SkillBook can also “countersign” verified releases

## Policy levels

* **Experimental:** passes schema only
* **Community Verified:** passes full pipeline
* **OpenClaw Verified:** pipeline + manual review + signed
